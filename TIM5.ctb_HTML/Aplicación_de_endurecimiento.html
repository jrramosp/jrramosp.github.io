Gestión de parches de software


Si bien algunas partes de las características del software están expuestas, una gran cantidad de ataques dependen de explotar errores en el software. Esto desencadena un comportamiento involuntario y extraño que puede poner en riesgo el sistema que ejecuta el software vulnerable. Estos tipos de vulnerabilidades se pueden solucionar mediante parches y actualizaciones de software que corrigen los errores que los atacantes explotan.
Como especialista en soporte de TI, es fundamental que te asegures de instalar actualizaciones de software y parches de seguridad de manera oportuna para defender los sistemas y redes de tu empresa. Las actualizaciones de software no solo mejoran los productos de software mediante la incorporación de nuevas funcionalidades y aumentos en el nivel de desempeño y estabilidad, también abordan las vulnerabilidades de seguridad.
Hay algunos errores de software que están presentes en la funcionalidad básica del software en cuestión. Esto significa que la vulnerabilidad no puede mitigarse deshabilitando el servicio vulnerable, eso no es bueno.
La mejor protección es tener un buen sistema y una buena política en tu empresa. El sistema debe comprobar, distribuir y verificar actualizaciones de software para la implementación del software. Este es un problema complejo si se piensa en una gran organización con muchas máquinas para administrar que ejecutan una variedad de productos de software.
Aquí es donde las herramientas de administración pueden hacer que esta tarea sea más accesible para ti. Soluciones como SCCM de Microsoft o las herramientas Puppet de Puppet Labs permiten a los administradores obtener una visión general de qué software está instalado en su inventario multisistema. Esto permite que un equipo de seguridad analice qué software específico y qué versiones están instaladas para comprender mejor el riesgo de software vulnerable en el inventario. Cuando se publican actualizaciones y se las envía al inventario, estas herramientas de informes pueden ayudar a garantizar que las actualizaciones se hayan aplicado. SCCM incluso tiene la capacidad de forzar la instalación de actualizaciones después de que haya transcurrido un plazo específico.
Los parches no solo son necesarios para el software, sino también para sistemas operativos y firmware que se ejecutan en dispositivos de infraestructura. Cada dispositivo tiene un código en ejecución que podría tener errores de software que podrían dar lugar a vulnerabilidades de seguridad desde routers, conmutadores, teléfonos hasta impresoras. Los proveedores de sistemas operativos suelen enviar parches relacionados con la seguridad con bastante rapidez cuando se descubre un problema. Por lo general, lanzarán correcciones de seguridad fuera de ciclo de actualizaciones típicas del SO para garantizar una solución oportuna debido a las implicaciones de seguridad. Pero para dispositivos integrados como equipos de red o impresoras, esto podría no ser lo usual.
Al aplicar actualizaciones, los dispositivos de infraestructura crítica deben abordarse con cuidado. Siempre existe el riesgo de que una actualización de software introduzca un nuevo error que puede afectar la funcionalidad de un dispositivo, o que el proceso de actualización salga mal y provoque una interrupción.

Políticas de aplicación


Como puedes ver, el software de aplicación puede representar una superficie de ataque bastante grande. Esto es especialmente cierto cuando se trata de un gran inventario de sistemas usados en toda una organización. Por lo tanto, es importante contar con algún tipo de políticas de aplicación implementadas. Estas políticas tienen dos propósitos. No solo definen los límites de lo que las aplicaciones pueden o no hacer, sino también ayudan a educar a la gente sobre cómo usar el software de forma más segura.
Ya vimos los riesgos que el software puede presentar debido a las vulnerabilidades de seguridad. Tiene sentido tener una política sobre la aplicación de actualizaciones de software de manera oportuna. Una recomendación o requisito común es solo brindar soporte o exigir la última versión de un software. Desde la perspectiva de soporte de TI, esto es importante porque las actualizaciones de software a menudo solucionarán los problemas que alguien más puede encontrar. Pero desde el lado de la seguridad, garantizar la última versión del software asegurará que se aplicaron todos los parches de seguridad y que la versión más segura está en uso.
Esto debe señalarse claramente en una política. Las personas tienden a ser bastante vagas a la hora de aplicar actualizaciones al software que usan mucho. Muchas veces, aplicar una actualización requiere reiniciar la aplicación, lo que puede resultar inconveniente y perjudicial para los usuarios. En general, es una buena idea rechazar clases de software de riesgo por medio de políticas. Cosas como software para compartir archivos y el software relacionado con la piratería tienden a estar estrechamente asociadas con infecciones de malware. Por lo general, tampoco tienen un uso comercial. Ni siquiera hablemos de las implicaciones legales de este tipo de software.
Comprender qué necesitan sus usuarios para hacer su trabajo te ayudará a dar forma a tu enfoque de las políticas y directrices de software. Si hay un caso de uso común para un cierto tipo de software, sería útil seleccionar una implementación de software específica y exigir el uso de esa solución. Esto permite reevaluar la solución más segura y beneficiarse de una instalación de software más uniforme. Recuerda, el quid de la cuestión es minimizar las superficies de ataque. Cada software que hace la misma cosa representa un conjunto diferente de posibles superficies de ataque que podrían tener una vulnerabilidad al acecho.
Ayudar a tus usuarios a realizar tareas recomendando un software específico o brindándole soporte a ese software permite lograr un entorno más seguro. También ayuda a los usuarios al poner a su disposición soluciones claras para realizar tareas. Si deseas emplear una solución de lista blanca de códigos binarios, también es importante definir una política sobre qué tipo de software se puede incluir en la lista blanca. Probablemente no sea necesario incluir videojuegos en la lista blanca, a menos que tu empresa sea un estudio de videojuegos, por supuesto. Estas políticas generalmente requieren algún tipo de caso de uso de negocios o justificación para evitar un montón de solicitudes personales de software.
Otra clase de software para el que quizás quieras definir políticas son las extensiones de navegador o complementos. Dado que muchos flujos de trabajo ahora residen exclusivamente dentro del navegador web, representan un vector potencial para el malware que a menudo se pasa por alto. Las extensiones que requieren acceso completo a los sitios web visitados pueden ser de riesgo, ya que el desarrollador de la extensión tiene el poder de modificar las páginas visitadas. Algunas extensiones pueden incluso enviar entradas de usuarios a un servidor remoto. Esto podría filtrar potencialmente información confidencial. Definir claramente las clasificaciones de extensiones y complementos de riesgo ayudará a proteger tus sistemas y brindará orientación a tus usuarios.
Pero las políticas no suelen ser suficientes para dotar a los usuarios de la información que necesitan para tomar decisiones informadas. Sus decisiones pueden afectar la seguridad de tu organización. Ahí es donde entran en juego la educación y la capacitación.
